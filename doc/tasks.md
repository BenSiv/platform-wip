## Completed (most recent first)

- 94: Evaluated a tamper-evidence/immutability layer for the ledger -- verified against the real code and real production access controls, not just the task's own original framing. Confirmed directly: `entity_event` has exactly two `UPDATE` statements in the whole codebase (`ledger.lua`'s `append_create`/`append_archive`), and both are immediate post-insert fixups completing a just-inserted row's own `entity_id`/`event_type` before any concurrent reader could observe the intermediate state -- not retroactive edits to an already-observable row. The append-only invariant genuinely holds in application code. The real threat is a direct database connection bypassing the app entirely (documented in `~/software/docs/platform-direct-database-connectivity.md`, found during task #99): only one such credential exists (the full-access `platform` MySQL user), gated by `roles/cloudsql.client` IAM + a Cloud SQL Auth Proxy tunnel, password in Secret Manager, zero public network exposure on the instance itself -- and a direct write is invisible to `entity_event` forever, confirmed the hard way when task #99's own testing accidentally deleted 2 live rows (recovered only via Cloud SQL's point-in-time clone, not any ledger-level protection). Recommendation if this is ever built: HMAC-sign each row (`row_signature = hmac(secret, event_id .. entity_id .. entity_type .. event_type .. field_changes .. created_at)`), reusing `auth.lua`'s own session-signing pattern (a locally-generated secret, `hmac.sha256`, never transmitted) rather than a new primitive -- and deliberately keep that signing secret out of Secret Manager, since the whole point is that it must stay unreachable to whoever holds the `platform-db-password` credential (a different access path: Auth Proxy + IAM, not container/host filesystem access). Plain hash-chaining alone was considered and rejected as the primary mechanism -- without a secret key, anyone with the write access this is meant to guard against can simply recompute the chain forward after an edit; HMAC signing is the meaningfully stronger version of the same idea at comparable cost. Not implementing now: the one demonstrated real-world failure to date was operator error (an over-privileged grant), not deliberate tampering, and Cloud SQL PITR already recovered it fully; adding signing now means a schema migration against live production data plus a real key-distribution and verification-job design for a risk that hasn't materialized and isn't yet a compliance requirement. Revisit if GLP/21 CFR Part 11 certification is ever pursued, or if a second direct-DB-write credential is ever created (widening who could tamper undetected beyond today's single, tightly IAM-gated one).
